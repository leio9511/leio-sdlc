# LEIO SDLC (v0.8.4)

LEIO SDLC is a multi-agent Software Development Life Cycle (SDLC) engine for OpenClaw. It automates planning, coding, reviewing, and merging through a fully isolated, artifact-driven pipeline.

**Note on Repository Structure:** This project actually contains a "Skill Kit" bundling two AgentSkills together:
1. `leio-sdlc`: The core automation engine.
2. `pm-skill`: The strict PRD authoring and initialization tool.

Both skills are deployed simultaneously using the `kit-deploy.sh` script.

## Core Architecture: Control Plane vs. Data Plane

As of v0.8.x, LEIO SDLC implements a strict separation of concerns to ensure the target codebase remains pristine during execution:

- **Data Plane (Working Directory)**: The target project's root folder (`--workdir`). This is where actual business code lives and where `git commit` / `git merge` operations happen. No SDLC runtime artifacts are stored here.
- **Control Plane (Global Run Directory)**: An isolated sandbox (`--global-dir`, default: `/root/.openclaw/workspace/.sdlc_runs_global`) that stores all AI runtime artifacts:
  - PR Contracts (Micro-slices)
  - `Review_Report.md` & `*.diff` files
  - `.coder_session` and state machine trackers
  - Forensic logs and quarantine states

### Configuration & Dynamic Merge

The framework uses a centralized configuration file located at `config/sdlc_config.json`.
- **Dynamic Updates**: During initialization, the engine merges missing keys from `sdlc_config.json.template` into the active config, ensuring seamless backward compatibility during framework upgrades.
- **`GLOBAL_RUN_DIR`**: The fallback path for the Control Plane if the `--global-dir` argument is omitted.

## Core Components (The Trio Model)

1. **The Planner (Micro-Slicing Engine)**:
   - Evaluates a Product Requirements Document (`PRD.md`).
   - Slices the PRD into sequential, dependency-ordered micro-PR contracts flatly stored in the Control Plane (`.sdlc_runs_global/<project_name>/<prd_name>/`).

2. **The Manager (Queue Polling Orchestrator)**:
   - Automated continuous dispatcher executing the State Machine (FSM).
   - Manages the **Review-Correction Loop (Yellow Path)** and the **Escalation Loop (Red Path)**.

3. **The Coder (TDD Closed-Loop Worker)**:
   - Bound by strict local TDD rules: must execute `./preflight.sh` within its sandbox and resolve errors before requesting a review.

4. **The Reviewer (Artifact-Driven Auditor)**:
   - Reads the actual `git diff` generated by the Coder.
   - Outputs a definitive `APPROVED` or `ACTION_REQUIRED` stamp.

## Testing & Sandboxing
The test suite utilizes a strict `test_tmp` isolation pattern. Tests explicitly pass a temporary path to `--global-dir` (e.g., `/tmp/mock_sdlc_global_$$`), ensuring mock artifacts never pollute production run records.

## Installation & Usage

### 1. Installation
Deploy both `leio-sdlc` and `pm-skill` locally using the kit installer:
```bash
./kit-deploy.sh
```

### 2. Standard Workflow (Strict Enforcement)

To use the SDLC engine, you **MUST** follow this strict linear process. You cannot manually write a PRD and feed it to the orchestrator directly.

**Step 1: PRD Authoring (`pm-skill`)**
Use `pm-skill` to safely scaffold and write the PRD template.
```bash
python3 ~/.openclaw/skills/pm-skill/scripts/init_prd.py --project <YourProject> --title "<Feature_Title>"
```
*Fill in the generated template with strict BDD acceptance criteria.*

**Step 2: Technical Audit (`Auditor`)**
Before any code is generated, the PRD must pass the architectural and structural audit.
```bash
python3 ~/.openclaw/skills/leio-sdlc/scripts/spawn_auditor.py --prd-file <Path_to_PRD.md> --workdir . --channel <your_channel>
```
*(You must resolve any `ACTION_REQUIRED` rejections until the Auditor returns `APPROVED`.)*

**Step 3: Engine Ignition (`Orchestrator`)**
Once approved by the Auditor, start the multi-agent pipeline:
```bash
python3 ~/.openclaw/skills/leio-sdlc/scripts/orchestrator.py --prd-file <Path_to_PRD.md> --workdir . --channel <your_channel>
```
*(Ensure your `GLOBAL_RUN_DIR` is configured in `sdlc_config.json` or pass `--global-dir` explicitly.)*
