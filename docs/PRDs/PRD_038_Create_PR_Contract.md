# PRD_038: Create PR Contract Engine (Unix Philosophy for LLM Queues)

## 1. Problem Statement
The current pipeline architecture relies on the Planner to manually create `PR_*.md` files via native filesystem tools (like `write`). This leads to "Leaky Abstractions" where the Planner might hallucinate file names (e.g., `PR-01.md` instead of `PR_001.md`), forget to prepend `status: open`, or write to the wrong directory. 
To adhere to the Unix Philosophy for LLMs (Entry Point Separation with Underlying Cohesion), we must provide the Planner with a single, atomic "Dealer" tool: `create_pr_contract.py`.

## 2. Solution Details: `scripts/create_pr_contract.py`

This script is the exclusive API the Planner will use to push tasks into the `leio-sdlc` Job Queue.

### 2.1 CLI Interface
- `--job-dir <path>` (Required): The directory of the current project/feature queue (e.g., `.sdlc/jobs/Feature_X/`).
- `--title <string>` (Required): A concise name for the PR (e.g., "DB_Schema", "Auth_API").
- `--content-file <path>` (Required): The path to a temporary file containing the raw PR contract text generated by the Planner.
- `--insert-after <prefix>` (Optional): Support dynamic reslicing. If provided (e.g., `003`), the script will append a letter suffix to insert the task while preserving the overall queue order (e.g., `003a`, `003b`).

### 2.2 Core Logic
1. **Directory Validation**: Ensure `--job-dir` exists. If not, create it.
2. **File Validation**: Ensure `--content-file` exists and is readable.
3. **Queue Indexing (Append vs Insert)**: 
   - *Default (Append)*: Scan `--job-dir` for existing PRs. Find the maximum base number (e.g., `003`). The new index is `004`.
   - *Insert (`--insert-after 003`)*: Scan for files matching `PR_003_*.md` (e.g., `PR_003_1`, `PR_003_2`). Find the highest sub-index. If none exist, the new index is `003_1`. If `003_1` exists, the new index is `003_2`.
   - *Lexical Sorting Magic*: In Python/Bash, `PR_003_1` comes after `PR_003` but before `PR_004`. This perfectly preserves execution order!
4. **Filename Generation**: Combine index and title into `PR_{index}_{title}.md` (e.g., `PR_004_Auth_API.md` or `PR_003_1_Auth_API.md`). Replace spaces with underscores.
5. **State Injection**: 
   - Read the contents of `--content-file`.
   - Prepend `status: open\n\n` to the top of the content.
6. **File Creation**: Write the finalized content to the new path inside `--job-dir`.
7. **Success Output**: Print `[PR_CREATED] <full_path_to_new_file>` and exit 0.

### 2.3 Refactoring for Cohesion (Optional but Recommended)
If `scripts/get_next_pr.py` and `scripts/update_pr_status.py` share logic (like scanning the job directory), Coder can extract this into `scripts/queue_utils.py` and import it across all three CLI endpoints.

## 3. Testing Strategy (TDD)
Create `scripts/test_create_pr_contract.sh`.

### 3.1 Test Cases
1. **Negative**: Missing `--job-dir`, `--title`, or `--content-file`. Assert `exit 1` and `[Pre-flight Failed]` or `argparse` usage error.
2. **Negative**: `--content-file` does not exist. Assert `exit 1`.
3. **Positive (First PR)**: 
   - Call script with empty `--job-dir`. 
   - Assert output contains `[PR_CREATED] .../PR_001_DB.md`.
   - Assert `cat PR_001_DB.md` starts with `status: open`.
4. **Positive (Increment)**: 
   - Call script again on same `--job-dir` with title "API".
   - Assert output contains `[PR_CREATED] .../PR_002_API.md`.
5. **Positive (Insert)**: 
   - Call script with `--insert-after 002` and title "Auth".
   - Assert output contains `[PR_CREATED] .../PR_002_1_Auth.md`.
   - Call script again with `--insert-after 002` and title "Login".
   - Assert output contains `[PR_CREATED] .../PR_002_2_Login.md`.

### 3.2 Preflight Integration
Append `bash scripts/test_create_pr_contract.sh` to `preflight.sh`.

## 4. Acceptance Criteria
- [ ] `scripts/test_create_pr_contract.sh` is written and passes all positive/negative edge cases.
- [ ] `scripts/create_pr_contract.py` automatically injects `status: open`.
- [ ] `scripts/create_pr_contract.py` automatically increments the `PR_xxx_` prefix.
- [ ] The script enforces strict role boundaries: it only creates, it cannot read the queue or update statuses.
- [ ] Running `./preflight.sh` successfully executes this new test suite.
