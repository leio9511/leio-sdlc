# PRD-057: PR Directory Isolation by PRD (Fix Global Queue Chaos)

## 1. Background & Problem Statement
Currently, the SDLC pipeline dumps all micro-PR contracts generated by the Planner into a single, global directory: `docs/PRs/`. 
The `orchestrator.py` engine blindly polls this root directory for any file containing `status: open`. If multiple projects (PRDs) are being processed concurrently, or if stale/failed PRs from previous runs are left behind, the Orchestrator will pick them up indiscriminately. This "Global Queue Chaos" causes cross-contamination, where the engine might attempt to apply code from PRD_A while executing the context of PRD_B.

## 2. Objectives
Enforce strict physical isolation for PR queues. Every PRD must have its own dedicated subdirectory inside `docs/PRs/`. The Orchestrator and Planner must be updated to respect and enforce this routing.

## 3. Functional Requirements

### 3.1 Dynamic Job Directory Computation
The "Job Directory" for any given project must be dynamically derived from the PRD file name.
- Example: If the PRD is `docs/PRDs/PRD_057_Isolation.md`, the Job Directory must be `docs/PRs/PRD_057_Isolation/`.
- Logic: Extract the base name of the PRD file (removing the `.md` extension) and append it to `docs/PRs/`.

### 3.2 Planner Updates (`scripts/spawn_planner.py`)
- Modify the default behavior of the `--out-dir` argument.
- If `--out-dir` is not explicitly provided, the Planner must compute the Job Directory from the `--prd-file` argument.
- The Planner must automatically create this directory (`os.makedirs(..., exist_ok=True)`) before writing the generated `PR_*.md` files.

### 3.3 Orchestrator Updates (`scripts/orchestrator.py`)
- The Orchestrator currently uses a hardcoded or CLI-provided `--job-dir` (defaulting to `docs/PRs`).
- **Update**: Override or set the `job_dir` variable by dynamically computing it from the required `--prd-file` argument.
- The Orchestrator's internal polling loop (State 1) and any calls to `get_next_pr.py` must strictly point to this isolated subdirectory.

## 4. Testing Strategy & Acceptance Criteria (TDD)
To ensure this is implemented correctly without breaking the engine, the Coder must update `scripts/test_orchestrator_fsm.sh` with the following deterministic sandbox scenarios:

### Scenario 1: Planner Isolation Assertions
- **Action**: Run the stubbed/mocked or real `spawn_planner.py` (if safe in sandbox) with `--prd-file docs/PRDs/Dummy_Project.md`.
- **Assertion**: Verify that the output files are placed exclusively in `docs/PRs/Dummy_Project/`.
- **Assertion**: Verify that no PR files leak into the root `docs/PRs/` directory.

### Scenario 2: Orchestrator Polling with Noise Injection
- **Action**: In the test sandbox, artificially create a "poisoned" environment:
  - Create a fake, open PR in the root: `docs/PRs/Poison_PR.md` (`status: open`).
  - Create a valid, open PR in the isolated dir: `docs/PRs/Target_Project/Target_PR.md` (`status: open`).
- **Action**: Run `orchestrator.py --prd-file docs/PRDs/Target_Project.md --max-runs 1`.
- **Assertion**: The Orchestrator MUST pick up `Target_PR.md` and transition its status to `closed`.
- **Assertion**: The Orchestrator MUST completely ignore `Poison_PR.md` (its status must remain `open`).

### Scenario 3: Missing Directory Graceful Sleep
- **Action**: Run `orchestrator.py --prd-file docs/PRDs/Empty_Project.md --max-runs 1` where the directory `docs/PRs/Empty_Project/` does not even exist yet.
- **Assertion**: The Orchestrator must NOT crash with a `FileNotFoundError`. It should gracefully handle the missing directory (treating it as an empty queue) and exit 0.

## 5. Rollout Plan
Once merged, a human operator will manually clean up any legacy `.md` files residing directly in the `docs/PRs/` root to finalize the migration to isolated queues.
